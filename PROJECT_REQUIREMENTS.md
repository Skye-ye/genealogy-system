## 二、业务场景描述

系统中管理多个**族谱**，每个族谱对应一个家族。

**族谱基本信息**：

- 谱名
- 姓氏
- 修谱时间
- 创建用户

**系统特性**：

- 支持多用户，每个用户都可以创建自己的族谱
- 用户可邀请其他用户一起为自己创建的族谱增加成员

**族谱中需记录**：

- 家族成员的基本信息：姓名、性别、生卒年、生平简介
- 成员间的血缘关系：父子 / 父女 / 母子 / 母女
- 成员间的婚姻关系

### 核心挑战

1. 成员之间的层级关系深度不确定。
2. 存在同名同姓但不同辈分的情况。
3. 需要快速查询某个成员的所有直系后代或追溯祖先。
4. 需要快速查询两个人名之间是否存在亲缘关系以及两者之间的亲缘链路。

## 三、任务要求

### 1. 简单应用界面（Application Development）

使用 **Python (Flask / Django)** 或 **.NET (WinForm / WPF / WebAPI)** 或其他开发工具实现一个基础图形化界面：

- **用户登录**：用户只能看到和操作自己创建的或者受邀的族谱。
- **用户管理**：用户采用注册制。
- **Dashboard**：显示家族总人数、男女比例。
- **族谱、成员管理**：实现增删改查、族谱能邀请他人参与创作、成员查找能支持人名的模糊查找。
- **树形预览**：以层级结构（或缩进列表）展示某分支的人物关系树状图。
- **人物祖先查询**：输入一个人物的 ID，以树状图显示出其父辈以上的祖先。
- **人物亲缘关系查询**：输入两个人的 ID，查询两者之间是否有亲缘关系连成的通路；如有请展示出来。

### 2. 数据库建模与规范化设计（Conceptual & Logical Design）

- **ER 图绘制**：识别实体（成员、家庭等）及其属性，标明联系类型（1:1, 1:N, M:N）。
- **关系模式转换**：将 ER 图转换为关系表，并说明其达到的范式级别（**建议至少满足 3NF 或 BCNF**）。
- **约束设计**：定义主键、外键，并编写 CHECK 约束（如：父亲的出生年份必须早于子女）。

### 3. 数据生成与导入导出（Data Engineering）

- **模拟数据生成**：编写 Python 脚本或 SQL 存储过程，生成
  - 至少 **10 个族谱**；
  - 其中至少有一个族谱拥有 **50,000 以上**的成员；
  - 整个系统中不少于 **100,000 条**模拟成员数据；
  - 要求每一个族谱内的成员至少与另一个成员有亲缘关系；
  - 单个族谱内模拟出至少 **30 代**人的传承关系。
- **导入导出**：演示如何使用 `LOAD DATA`（MySQL）或 `COPY`（PostgreSQL）将 CSV 数据批量导入数据库，并导出某分支的备份文件。

### 4. SQL 核心功能编写（DML & Query）

编写满足以下需求的 SQL 语句（**每个需求用单个 SQL 实现**）：

1. **基本查询**：给定一个成员 ID，查询其配偶及所有子女。
2. **递归查询（核心）**：编写一个递归公用表表达式（**Recursive CTE**），输入成员 A 的 ID，输出其向上追溯的所有历代祖先。
3. **统计分析**：统计某个家族中平均寿命最长的一代人（辈分）。
4. 查询所有年龄超过 50 岁（假设当前日期减去出生日期）、且没有配偶的男性成员。
5. 找出家族中"出生年份"早于该辈分（代）平均出生年份的所有成员。

### 5. 物理优化与索引设计（Physical Design）

- **索引策略**：针对"根据姓名模糊查询"和"根据父节点 ID 查询子节点"的需求，设计合适的索引。
- **性能对比**：记录在有无索引情况下，执行"查询某曾祖父的所有曾孙（四代查询）"的执行时间差异，并提交**执行计划（EXPLAIN）分析**。

## 六、相关技术背景（来自前序章节）

为完成本项目，可能需要使用以下数据库服务器端开发能力：

- **PL/pgSQL**：openGauss / PostgreSQL 中的过程语言（变量、IF/ELSE、WHILE、RETURN）。
- **UDT（User Defined Type）**：组合类型、枚举类型、范围类型、全新类型。
- **UDF（User Defined Function）**：标量函数、表值函数；`CREATE FUNCTION`。
- **存储过程（Stored Procedure）**：`CREATE PROCEDURE` + `CALL`。
- **触发器（Trigger）**：BEFORE / AFTER / INSTEAD OF；可用于审计、复杂业务规则、系统事件处理、完整性约束加强。
- **数据库 API**：ODBC / OLE / JDBC / ADO.NET 等连接方式。
